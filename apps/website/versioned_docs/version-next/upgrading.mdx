# Upgrading to v10

DayPicker v10 removes the APIs that were deprecated in v9.

:::note Easy upgrade path
If your app is already on v9 and does not rely on deprecated APIs, the upgrade
is mostly mechanical.

The main behavior change outside the removed APIs is that omitting
`navLayout` now behaves like `navLayout="after"`.
:::

If you are upgrading from v8 or earlier, first follow the
[v9 upgrade guide](/upgrading) and remove the v9 deprecations there.

## 1. Upgrade the package

```bash npm2yarn
npm install react-day-picker@next
```

## 2. Replace removed navigation props

| Removed prop | Use instead |
| --- | --- |
| `fromMonth` | [`startMonth`](./api/interfaces/PropsBase.md#startmonth) |
| `fromYear` | [`startMonth`](./api/interfaces/PropsBase.md#startmonth) with the first month of that year |
| `toMonth` | [`endMonth`](./api/interfaces/PropsBase.md#endmonth) |
| `toYear` | [`endMonth`](./api/interfaces/PropsBase.md#endmonth) with the last month of that year |
| `fromDate` | [`hidden`](./api/interfaces/PropsBase.md#hidden) with a `before` matcher, and optionally [`startMonth`](./api/interfaces/PropsBase.md#startmonth) |
| `toDate` | [`hidden`](./api/interfaces/PropsBase.md#hidden) with an `after` matcher, and optionally [`endMonth`](./api/interfaces/PropsBase.md#endmonth) |

```diff
<DayPicker
- fromYear={2010}
- toYear={2021}
+ startMonth={new Date(2010, 0)}
+ endMonth={new Date(2021, 11)}
/>
```

```diff
<DayPicker
- fromDate={new Date(2010, 11, 3)}
+ startMonth={new Date(2010, 11)}
+ hidden={{ before: new Date(2010, 11, 3) }}
/>
```

```diff
<DayPicker
- toDate={new Date(2010, 11, 3)}
+ endMonth={new Date(2010, 11)}
+ hidden={{ after: new Date(2010, 11, 3) }}
/>
```

## 3. Replace removed focus and event props

| Removed prop | Use instead |
| --- | --- |
| `initialFocus` | [`autoFocus`](./api/interfaces/PropsBase.md#autofocus) |
| `onWeekNumberClick` | A custom [`WeekNumber`](./api/functions/WeekNumber.md) component |
| `onDayKeyUp` | A custom [`DayButton`](./api/functions/DayButton.md) component |
| `onDayKeyPress` | A custom [`DayButton`](./api/functions/DayButton.md) component |
| `onDayPointerEnter` | A custom [`DayButton`](./api/functions/DayButton.md) component |
| `onDayPointerLeave` | A custom [`DayButton`](./api/functions/DayButton.md) component |
| `onDayTouchCancel` | A custom [`DayButton`](./api/functions/DayButton.md) component |
| `onDayTouchEnd` | A custom [`DayButton`](./api/functions/DayButton.md) component |
| `onDayTouchMove` | A custom [`DayButton`](./api/functions/DayButton.md) component |
| `onDayTouchStart` | A custom [`DayButton`](./api/functions/DayButton.md) component |

For week-based selections, follow the
[week selection example](./guides/custom-selections.mdx#week-numbers-and-starts).

For extra DOM event handlers on each day, compose around the default
[`DayButton`](./api/functions/DayButton.md) in the
[Custom Components guide](./guides/custom-components.mdx).

## 4. Replace removed aliases

| Removed API | Use instead |
| --- | --- |
| `formatMonthCaption` | [`formatCaption`](./api/functions/formatCaption.md) |
| `formatYearCaption` | [`formatYearDropdown`](./api/functions/formatYearDropdown.md) |
| `labelDay` | [`labelDayButton`](./api/functions/labelDayButton.md) |

If you already migrated to the v9 names, there is nothing to change here.

## 5. Update custom components

The deprecated `components.Button` entry was removed in v10.

Use
[`PreviousMonthButton`](./api/functions/PreviousMonthButton.md) and
[`NextMonthButton`](./api/functions/NextMonthButton.md) instead.

```diff
<DayPicker
  components={{
-   Button: MyButton
+   PreviousMonthButton: MyPreviousButton,
+   NextMonthButton: MyNextButton
  }}
/>
```

## 6. Update `classNames` and `styles` keys

The deprecated compatibility keys were removed from `classNames` and `styles`.
Use only the current keys from
[`UI`](./api/enumerations/UI.md),
[`DayFlag`](./api/enumerations/DayFlag.md),
[`SelectionState`](./api/enumerations/SelectionState.md), and
[`Animation`](./api/enumerations/Animation.md).

Common migrations:

| Removed key | Use instead |
| --- | --- |
| `table` | `month_grid` |
| `nav_button` | `button_previous` and `button_next` |
| `day_selected` | `selected` |
| `day_disabled` | `disabled` |

## 7. Update imports

These imports were removed in v10:

- `react-day-picker/jalali` - use `@daypicker/persian`
- `react-day-picker/buddhist` - use `@daypicker/buddhist`
- `react-day-picker/ethiopic` - use `@daypicker/ethiopic`
- `react-day-picker/hebrew` - use `@daypicker/hebrew`
- `react-day-picker/hijri` - use `@daypicker/hijri`
- `react-day-picker/persian` - use `@daypicker/persian`
- `amET` from `react-day-picker/locale` - use `amET` from
  `@daypicker/ethiopic`
- `enUSJalali` from `react-day-picker/locale` - use `enUSJalali` from
  `@daypicker/persian`
- `faIRJalali` from `react-day-picker/locale` - use `faIRJalali` from
  `@daypicker/persian`
- Deprecated aliases from `types/deprecated`

The calendar packages are now published separately from the main React package.
Install the add-on package you use alongside `@daypicker/react@next`. For
example:

```bash npm2yarn
npm install @daypicker/react@next @daypicker/persian@next
```

## 8. Review the new `navLayout` default

In v10, omitting [`navLayout`](./api/interfaces/PropsBase.md#navlayout) is
equivalent to `navLayout="after"`.

Check:

- Navigation button placement
- Focus order and keyboard navigation
- Tests or snapshots that assert DOM order or `data-nav-layout`

If you want the behavior to stay explicit across upgrades, set `navLayout`
directly to `"after"` or `"around"`.

## 9. Audit copied wrappers

If your app uses a copied wrapper around DayPicker, audit that local file after
upgrading.

This is especially important for wrappers that still reference removed props
like `initialFocus`, `fromMonth`, `toMonth`, `fromYear`, or `toYear`, or old
class name keys like `table`, `nav_button`, `day_selected`, and
`day_disabled`.
